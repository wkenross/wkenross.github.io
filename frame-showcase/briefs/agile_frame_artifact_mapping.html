FRAME Artifact Mapping To Agile

FRAME artifacts are practical working documents. The Agile workflow translates them into standard Agile and Jira language without throwing away the Markdown source of thought.

Mapping

FRAME artifact Agile equivalent Jira target
PlanMe Epic or feature brief Epic
StartMe Initiation note / discovery brief Epic description or discovery task
BuildMe User stories and acceptance criteria Story
CodeMe Engineering task plan Task or sub-task
RunMe Definition of Done, runbook, validation Story test evidence / linked doc
TrainMe Learning and adoption guide Training story / enablement artifact
CertifyMe Credential/evidence map Certification backlog item
TaskMe Sprint focus list Sprint backlog
README Product overview Project summary / landing page

Public Artifact Labels

The HTML factory now supports a separate artifact_type metadata field. This lets FRAME keep local operator names such as PlanMe, RunMe, and README while public pages present familiar artifact labels.

Artifact Type Reader Signal
overview Product or service landing page
architecture Boundaries, interfaces, data contracts, risks, and decisions
business-plan Product brief, personas, value hypothesis, metrics, and proof status
roadmap Sequencing, milestones, horizons, and tradeoffs
backlog Epics, user stories, acceptance criteria, sprint candidates, and risks
runbook Operating steps, validation, launch checks, rollback, and support evidence
research Source-backed discovery, vendor notes, assumptions, and open questions
tool Workbench, fixture, or reusable service artifact

For the B2B eCommerce Adapter pilot, this means the business plan, architecture page, agile backlog, research pack, workbench, and prototype lanes can read as standard product-management artifacts without breaking the Markdown-first FRAME workflow.

Translation Flow


flowchart TD
    readme[README: product overview] --> planme[PlanMe: why and scope]
    planme --> epic[Jira Epic]
    buildme[BuildMe: user-visible behavior] --> story[Jira Story]
    codeme[CodeMe: implementation details] --> task[Jira Task]
    runme[RunMe: validation steps] --> dod[Definition of Done]
    trainme[TrainMe: learning loop] --> cert[Certification evidence]
    taskme[TaskMe: current focus] --> sprint[Sprint backlog]
    epic --> sprint
    story --> sprint
    task --> sprint
    dod --> review[Sprint review]
    cert --> review

Rule

Markdown remains the thinking surface. Jira becomes the visible planning and review board. The two should reinforce each other instead of competing.

Definition Of Done

A FRAME Agile story is done when:

  • the relevant docs or code are updated
  • the command or workflow can be run
  • validation or evidence exists
  • risks and manual steps are documented
  • the work can be explained in Agile language

Product Risk Analysis

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.
