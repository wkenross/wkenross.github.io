This document is written as public-safe FRAME showcase material. It describes the operating pattern without publishing passwords, browser profiles, customer workbooks, private tenant links, cookies, tokens, or private account state.

Overview

FRAME is a Windows automation framework organized around reproducible work lanes. Each lane has a named Windows virtual desktop, a VS Code workspace, a machine-local Edge remote-debugging profile, and workflow-specific input/output folders. The result is a practical operating model for running several automation streams on one laptop without mixing browser sessions, Codex chat context, Outlook windows, or source material.

The current Dell Inspiron 15 Windows 11 workstation uses four principal desktop lanes:

Desktop Workspace Role Edge port
frame_install Install.code-workspace framework installation, configuration, and documentation control 9234
frame_update fr_update.code-workspace local-first revenue/update framework source review 9236
Resumes resumes.code-workspace resume, job search, Outlook Classic, and application workflow 9230
Search_update sr_update.code-workspace Search-side publication staging, GitHub-safe intake, wb_html, and wb_github 9235

Why Virtual Desktops Matter

The framework uses virtual desktops as lightweight operating boundaries. The goal is not only a cleaner screen. The goal is consistent state.

Each desktop keeps the active workflow close to its own browser profile, VS Code workspace, open documents, and operator context. This reduces accidental cross use between private source review, public publication staging, resume/job search work, and install configuration.

The approach is especially useful when the same Windows user account needs to run several automations that all use Edge, VS Code, PowerShell, Python, and Codex. Instead of relying on memory to know which browser window belongs to which task, the launcher binds each workflow to a named desktop and a unique debug port.

Windows Machine Pattern

The machine-level pattern is intentionally repeatable:

  • OneDrive carries project folders, workspace files, documentation, intake

    • notes, and publication candidates.

  • Machine-local folders carry browser profiles, WebDriver, Python virtual

    • environments, Office identity state, and other runtime state that should not be synced between computers.

  • PowerShell orchestrators create or verify desktops, launch VS Code

    • workspaces, start Edge with a unique profile and remote-debugging port, and write canvas reports.

  • Python scripts handle deeper workflow operations where structured processing,

    • browser automation, Excel, SQL, or generated artifacts are needed.

  • Smoke tests verify the desktop canvas and Edge CDP ports before the workflow

    • is treated as ready.

This lets another Windows 11 laptop be brought online from synced docs and scripts while still rebuilding private runtime state locally.

iPhone And Mobile Integration

FRAME also includes an iPhone/mobile context loop. The phone is treated as a capture and handoff surface rather than as the main execution environment.

Typical mobile inputs include dictated notes, requirements snippets, quick captures, review prompts, and context packets. These are staged through mobile-safe Markdown and text files under the install and Codex Mobile folders. The Windows desktop lanes then pick up those inputs through the appropriate VS Code workspace.

This keeps mobile capture useful without mixing it directly into browser profiles, credentials, or live automation state. The phone can add context; the Windows lane runs the workflow.

Desktop Builder

The four-desktop launcher is:


fr_install\scripts\ps1\Invoke-FrameDesktopBuilder.ps1

The Start Menu and Desktop shortcut are created by:


fr_install\scripts\ps1\Install-FrameDesktopBuilderShortcut.ps1

Normal launch mode is:


pwsh -NoProfile -ExecutionPolicy Bypass -File "$env:OneDrive\projects\fr_install\scripts\ps1\Invoke-FrameDesktopBuilder.ps1" -NonInteractive -SkipHealthy

Review mode is:


pwsh -NoProfile -ExecutionPolicy Bypass -File "$env:OneDrive\projects\fr_install\scripts\ps1\Invoke-FrameDesktopBuilder.ps1" -Canvas -StrictCanonical

The builder writes a canvas report showing the desktop inventory, expected canonical desktops, legacy or extra desktops, and Edge CDP readiness.

Cleanup And Rebuild

Windows virtual desktops cannot be reduced to zero. The safe reset model keeps or creates one frame_install desktop, removes the other FRAME desktop lanes, and then lets the operator reboot before running the desktop builder again.

The reset script is:


fr_install\scripts\ps1\Reset-FrameDesktopCanvas.ps1

Dry-run review:


pwsh -NoProfile -ExecutionPolicy Bypass -File "$env:OneDrive\projects\fr_install\scripts\ps1\Reset-FrameDesktopCanvas.ps1"

Apply after saving work:


pwsh -NoProfile -ExecutionPolicy Bypass -File "$env:OneDrive\projects\fr_install\scripts\ps1\Reset-FrameDesktopCanvas.ps1" -Apply -CloseLaneEdge

The script leaves VS Code open by default so unsaved editor buffers are not lost. Add -CloseVSCode only after saving work.

After reset:

  1. Reboot Windows.
  2. Launch FRAME Desktop Builder from Start or the taskbar.
  3. Run the canvas check to confirm the four canonical desktops and ports are

    4. back in place.

Publication Lane

The same desktop model supports publication. Private source review happens in the local-first frame_update lane. Public-safe Markdown is promoted into the Search-side publication lane, where wb_html converts reviewed Markdown to HTML and wb_github packages the generated site material for GitHub Pages.

That separation is important. It keeps private working material out of the publication package, while still making the final public site reproducible from plain files and scripted build steps.

Review Checklist

  • No secrets, cookies, tokens, passwords, private browser profiles, or OAuth

    • state are copied between machines.

  • No customer workbook data or private tenant URLs are included in public

    • material.

  • Each desktop uses its own Edge profile and remote-debugging port.
  • The canvas report shows only frame_install, frame_update, Resumes, and

    • Search_update before normal operation.

  • Mobile/iPhone inputs enter through reviewed Markdown/text handoff files.
  • Publication candidates move from fr_install or workflow docs into

    • wb_html, then through wb_github, after public-safety review.
