FRAME Delivery Roadmap

Updated: 2026-06-16

Overview

This roadmap shows how the FRAME automation framework is being managed as a technical project-management program. It complements the project-plan overview with delivery phases, release intent, Jira-backed evidence, and completion artifacts.

The roadmap is public-safe: it avoids private local paths, credentials, browser-session details, private job-application material, and account-specific configuration.

Roadmap

Phase Stage Target Status Evidence
Phase 0 Vision Define the FRAME technical project-management operating model. Done PlanMe, TrainMe, stage-gate model
Phase 1 V0 Scope Create registers, templates, and weekly status reporting. Done project, milestone, risk, dependency, and decision artifacts
Phase 2 P1 Build Seed Jira and publish the public-safe project-plan page. Done Jira issue range SCRUM-30 through SCRUM-35; project-plan URL
Phase 3 P1 Build Add roadmap, release, Jira status, RAID, milestone, and completion evidence artifacts. In Progress delivery artifact reports
Phase 4 P2 Handoff Generate delivery artifacts through PowerShell orchestration and Python operation scripts. In Progress repeatable delivery-artifact generator
Phase 5 Run Maintain weekly project status and public-safe updates. Next recurring reports and reviewed publish inputs

B2B Adapter Future-Proofing Roadmap

The procurement adapter prototype should evolve through four visible slices:

Slice Purpose Evidence
Data syndication Model product identity, UOM, price, eligibility, safety documents, and account context as governed data. catalog/data-contract notes and fixture rules
Automation / agentic AI Define where AI may summarize, classify, recommend, or draft, and where human approval is required. AI guardrail notes and exception-summary examples
Evolving standards Track cXML, XML, OCI, EDI, hosted files, REST/OpenAPI, event/AsyncAPI, and provider-specific APIs. standards matrix and provider-lane assumptions
Monitoring maturity Compare API Fortress/Sauce Labs-style API testing, Postman, Playwright/APIRequest, contract testing, and platform-native monitoring. monitoring-options comparison and launch-health dashboard spec

Product Risk Analysis

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.

Publication rule: examples use public-safe or synthetic data only. They do not connect to live websites, mailboxes, CRM systems, marketplace accounts, or procurement tenants.
