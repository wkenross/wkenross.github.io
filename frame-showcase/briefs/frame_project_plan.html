FRAME Automation Project Plan

Updated: 2026-06-16

Overview

FRAME is managed as a review-first automation program. The operating model combines VS Code, Codex, PowerShell orchestration, Python operation scripts, mobile intake, Markdown artifacts, structured CSV/JSON records, and Jira-backed Agile visibility.

The goal is simple: keep automation useful, inspectable, and ready for handoff. The project plan makes the active work visible across workflows without publishing private local paths, credentials, job-application details, or account-specific browser state.

Project Management Stack

Layer Tooling Purpose
Intake iPhone notes, mobile Codex/Copilot/ChatGPT, fr_mobile folders Capture recommendations and requirements while away from the desktop.
Planning VS Code, Markdown PlanMe/TrainMe/RunMe/CodeMe/CertifyMe Explain intent, steps, implementation notes, and learning path.
Execution PowerShell orchestration scripts and Python operation scripts Run repeatable workflows and generate reports.
Records CSV registers and JSON seed files Track projects, milestones, risks, dependencies, decisions, and Jira inputs.
Visibility Jira Cloud through the Agile workflow Maintain epics, stories, sprint candidates, and status evidence.
Publishing wb_html and GitHub Pages Share sanitized project-plan and showcase pages.

Stage-Gate Model

FRAME projects move through a lightweight technical project-management model:

Stage Meaning Evidence
Vision Define the problem, value, users, constraints, and success signal. project brief or PlanMe
V0 Scope Define the smallest useful version and acceptance checks. milestone plan and risk review
P1 Build Create working scripts, docs, data, or controlled automation. working output and status report
P2 Handoff Make the workflow repeatable and explainable. RunMe, TrainMe, CodeMe, reports
Run Operate, monitor, improve, and route new intake. weekly status and backlog updates

Jira-Backed Evidence

The Agile workflow uses Jira as the visible planning board while Markdown and structured files remain the durable source of reasoning.

Current public-safe issue ranges:

Jira Range Meaning
SCRUM-5 through SCRUM-23 Initial Agile project-management operating system
SCRUM-24 through SCRUM-29 Search and product-owner toolkit implementation
SCRUM-30 through SCRUM-35 Technical project-management operating model

Jira is used for epics, stories, sprint selection, and status evidence. It does not replace the workflow documents, scripts, or project registers.

Intake And Update Workflow

New project-plan inputs can come from two places.

Mobile Intake

  1. Capture the idea or requirement on iPhone.
  2. Save it into the mobile intake workflow as a recommendation or requirement.
  3. The VS Code operator reviews the file with Codex.
  4. Codex routes the item to the owning workflow.
  5. The owning PlanMe/backlog/register/report is updated.
  6. The source input is marked processed after integration.

VS Code Workspace Intake

  1. A workflow operator discovers a new task, risk, milestone, decision, or

    2. publishing need while working in VS Code.

  2. Codex updates the local workflow artifacts.
  3. If the work needs visible Agile tracking, a reviewed JSON seed is prepared

    4. for Jira.

  4. Status reports and project-plan views are regenerated.
  5. Public-safe artifacts are staged through the publishing workflow only after

    6. review.

Client View

For clients, this plan demonstrates the operating discipline behind FRAME:

  • human-reviewed automation
  • explicit stage gates
  • visible risks and decisions
  • repeatable scripts
  • clear handoff docs
  • Jira-backed delivery visibility
  • mobile-to-desktop requirement capture
  • public-safe publishing controls

The client-facing story is not that automation runs unattended. The story is that automation makes better decisions easier to review, repeat, and explain.
