Desktop Builder And Canvas Readiness

Purpose

FRAME uses Windows virtual desktops as workflow lanes. The desktop builder prepares the screen, VS Code workspace, browser profile, and runtime checks for each lane so the operator can move between related workstreams without mixing state.

Lane Pattern

A FRAME workstation commonly uses lanes such as:

Lane Role
install framework setup, machine configuration, publication control
update local-first revenue and update workflow review
resumes job search, resume tailoring, and application workflow
search/update public-safe Search, SQL, BI, and website staging
for-hire packaged automation demonstrations and customer-facing materials

Each lane can have:

  • a named Windows desktop
  • a VS Code workspace
  • a dedicated Edge profile directory
  • a unique browser debug port
  • a small readiness report
  • a set of lane-specific docs and launch commands

Canvas Report

The canvas report answers practical questions:

  • which desktops are present
  • which lanes are expected
  • which Edge debug ports are reachable
  • whether the right workspace folders are available
  • whether a reset or reboot is recommended

The report is useful after a new machine install, after a reboot, and before an extended automation session.

Reset Philosophy

The reset workflow should be conservative:

  • preserve at least one usable Windows desktop
  • close lane-specific Edge windows only when requested
  • leave VS Code open unless the operator explicitly approves closing it
  • write a report before and after the reset
  • recommend reboot when Windows desktop state is stale

This protects unsaved work and keeps virtual desktop cleanup from becoming a destructive operation.

Publication Use

The desktop builder is a strong public showcase because it demonstrates that FRAME is more than isolated scripts. It is a repeatable operating environment: files, workspaces, browser automation, review docs, and publication lanes all open in a consistent structure.

Public documentation should explain the model and show sanitized command patterns. It should not expose raw window inventories, private workspace names that are not meant for publication, browser profile paths, or active account state.

Acceptance Checks

  • each lane opens the intended workspace
  • each lane uses a separate browser profile and debug port
  • the canvas report is readable by a future operator
  • reset scripts have dry-run behavior
  • reboot and relaunch steps are documented
  • public examples are sanitized before publication

Product Risk Analysis

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.
