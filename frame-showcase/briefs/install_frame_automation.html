Install FRAME Automation

Purpose

This page introduces the installation lane for the FRAME automation framework. It is intended for reviewers who want to understand how the Windows 11 laptop, VS Code workspaces, OneDrive project folders, Edge browser automation, and FRAME scripts fit together.

What The Install Workspace Does

The install workspace organizes the machine setup and follow-on configuration needed to run FRAME workflows on a new laptop.

It covers:

  • Windows 11 workstation preparation
  • PowerShell and Python setup
  • VS Code workspace launch configuration
  • OneDrive project synchronization
  • Edge browser profile and WebDriver support
  • FRAME folder placement for local and OneDrive modes
  • publication support for GitHub Pages and the FRAME showcase

Workstation Pattern

FRAME runs best as a workstation pattern, not just a folder of scripts. Each Windows machine is prepared to open several named workflow desktops. A typical setup uses four or five desktops, each dedicated to one automation stream. The desktops can be spread across multiple monitors so the operator can watch orchestrations, browser state, VS Code output, and workflow notes without mixing unrelated work.

Each desktop opens a focused VS Code workspace. The workspace includes the core frame folder and the supporting workflow folders for that lane. This gives Codex, PowerShell, Python, and browser automation the right local context while keeping shared FRAME helpers available across workflows.

The install scripts prepare the Windows application layer for that pattern:

  • required Windows applications and command-line tools
  • PowerShell and Python runtime checks
  • VS Code workspace settings and Codex handoff files
  • Edge and Chrome favorites/bookmark preparation
  • Edge remote-debugging profiles for workflow automation
  • WebDriver and PATH checks
  • OneDrive, iCloud, and mobile handoff folders
  • Outlook Classic and Office readiness where a workflow needs them

Manual sign-in, MFA, and account recovery remain operator actions. The scripts prepare applications and verify readiness; they do not copy private browser sessions, cookies, tokens, or credentials.

Browser And Mobile Continuity

FRAME uses a common browser identity strategy across desktops, machines, and iPhone. Edge and Chrome should present consistent favorites, bookmarks, and account menus so the same workflow links are available whether the operator is on the install desktop, the resume desktop, another Windows machine, or an iPhone review session.

Automation profiles remain machine-local. Each workflow lane recreates its own debug profile and port on the local workstation. This preserves repeatable automation behavior without publishing or syncing private browser profile folders.

The iPhone integration extends the same operating model. Mobile files and Shortcuts let the operator review key PlanMe and RunMe docs, dictate new requirements, and save Markdown drafts into a mobile requirements inbox. A Windows Codex session later intakes those drafts, inspects the live project files, and merges the requirement into the appropriate workflow.

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.
