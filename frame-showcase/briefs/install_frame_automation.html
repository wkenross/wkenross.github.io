Install FRAME Automation

Bootstrap Quick Menu

Use this when you are setting up a fresh Windows user or a new FRAME laptop.

  1. Sign in first. Log into the Windows user, open Edge, and sign in to the Microsoft account that can reach OneDrive.
  2. Download the full folder. From OneDrive live, download Services > Install into Downloads\Install. Do not download only the bootstrap folder for the normal path.
  3. Start the bootstrap. Open Downloads\Install\bootstrap\START_HERE_KR.md, then run COPY_PASTE_KR_BOOTSTRAP.txt or RUN_KR_BOOTSTRAP.cmd.

What happens next: first-login bootstrap starts the service router and runs user readiness checks. Full framework install/config continues through PostBootstrapContinuation; the runbook tells the operator when to start that second step. It is not a silent background install.

Purpose

This page introduces the installation lane for the FRAME automation framework. It is intended for reviewers who want to understand how the Windows 11 laptop, VS Code workspaces, OneDrive project folders, Edge browser automation, and FRAME scripts fit together.

What The Install Workspace Does

The install workspace organizes the machine setup and follow-on configuration needed to run FRAME workflows on a new laptop.

It covers:

  • Windows 11 workstation preparation
  • PowerShell and Python setup
  • VS Code workspace launch configuration
  • OneDrive project synchronization
  • Edge browser profile and WebDriver support
  • FRAME folder placement for local and OneDrive modes
  • publication support for GitHub Pages and the FRAME showcase

Workstation Pattern

FRAME runs best as a workstation pattern, not just a folder of scripts. Each Windows machine is prepared to open several named workflow desktops. A typical setup uses four or five desktops, each dedicated to one automation stream. The desktops can be spread across multiple monitors so the operator can watch orchestrations, browser state, VS Code output, and workflow notes without mixing unrelated work.

Each desktop opens a focused VS Code workspace. The workspace includes the core frame folder and the supporting workflow folders for that lane. This gives Codex, PowerShell, Python, and browser automation the right local context while keeping shared FRAME helpers available across workflows.

The install scripts prepare the Windows application layer for that pattern:

  • required Windows applications and command-line tools
  • PowerShell and Python runtime checks
  • VS Code workspace settings and Codex handoff files
  • Edge and Chrome favorites/bookmark preparation
  • Edge remote-debugging profiles for workflow automation
  • WebDriver and PATH checks
  • OneDrive, iCloud, and mobile handoff folders
  • Outlook Classic and Office readiness where a workflow needs them

Browser And Mobile Continuity

FRAME uses a common browser identity strategy across desktops, machines, and iPhone. Edge and Chrome should present consistent favorites, bookmarks, and account menus so the same workflow links are available whether the operator is on the install desktop, the resume desktop, another Windows machine, or an iPhone review session.

Automation profiles remain machine-local. Each workflow lane recreates its own debug profile and port on the local workstation. This preserves repeatable automation behavior without publishing or syncing private browser profile folders.

The iPhone integration extends the same operating model. Mobile files and Shortcuts let the operator review key PlanMe and RunMe docs, dictate new requirements, and save Markdown drafts into a mobile requirements inbox. A Windows Codex session later intakes those drafts, inspects the live project files, and merges the requirement into the appropriate workflow.

New-User Bootstrap Pattern

The current FRAME bootstrap pattern is intentionally human-first for identity and script-first for installation.

The operator signs in to the new Windows account, opens Edge, signs in to the Microsoft account that has access to the FRAME OneDrive material, and verifies that the cloud Services > Install folder is reachable. The scripts do not copy browser profiles, cookies, MFA sessions, OneDrive tokens, or saved passwords from another Windows user.

After identity is established, download the full Services > Install folder from OneDrive live to the new user's Downloads\Install folder. Downloading only the bootstrap folder is faster, but it leaves the new account without the service docs, router scripts, reports folder, bootstrap helpers, and validation files that Codex and PowerShell need when something goes wrong. The full folder download is the recommended default.

The first file for a clean current bootstrap is:


Downloads\Install\bootstrap\START_HERE_KR.md

That runbook points to the matching copy/paste command and command helper in the same bootstrap folder. The command locates the downloaded Install service, runs the service router, performs first-login checks, and records reports under the downloaded or synced service folder.

Bootstrap is the first step, not the whole install by itself. Framework install/config continues through PostBootstrapContinuation after the first-login checks and source readiness checkpoint pass. The runbook tells the operator when to start that second command; it is not a silent background install.

Codex or ChatGPT is useful at this stage as an operator assistant over the downloaded folder: it can read the runbook, inspect reports, explain failures, and choose the next service operation. It should not replace the runbook with a free-form "figure it out" install. The scripts remain the source of truth for machine changes.

Bootstrap Launcher Helpers

Use these helpers when bringing up a fresh Windows account.

Step Link or local launcher What it does
1 Open OneDrive online Sign in with the Microsoft account that can reach Services > Install, then download only that folder if desktop sync is not ready yet.
2 FRAME Copy Synced Install To Downloads.cmd If OneDrive desktop sync already contains Services\Install, this local launcher copies it to %USERPROFILE%\Downloads\Install and opens bootstrap\START_HERE_KR.md.
3 Downloads\Install\bootstrap\START_HERE_KR.md Follow the current KR runbook and copy/paste the bootstrap command from the downloaded service kit.

The public install page cannot directly run local PowerShell, unzip private OneDrive folders, or read a user's cloud files. That boundary is intentional. The page provides the official links and instructions; local launchers in the OneDrive or Services root handle local file moves after the signed-in user has made the folder available on the machine.

Install these core assistants first when a machine is missing them: Download VS Code, Download ChatGPT, Python for Windows, Microsoft 365 apps, and PowerShell for Windows.

Service Command Surface

Install operations are launched through a small service router so operators do not have to remember individual script paths. Important operations include:

Operation Purpose
PathConfigStatus Show resolved service, project, report, and bootstrap roots.
PrerequisiteCheck Verify PowerShell, Python, Git, winget, VS Code, Edge, WebDriver, and PATH readiness.
FirstLoginBootstrap Configure and validate the fresh Windows user after the full Install folder is downloaded.
PostBootstrapContinuation Continue after first-login bootstrap and publish the fixed service kit when checks pass.
KRAccountReset Reset the current clean install-test account lane from an administrator context.
DesktopBuilderReadiness Check resource state and desktop-builder readiness before opening workflow lanes.

The practical readiness rule is that FRAME should distinguish "required to operate" from "preferred latest." A machine can be usable while still carrying yellow follow-up items, such as an available PowerShell update, as long as the running scripts do not depend on the newer version.

Bootstrap Configuration Contract

After Edge/OneDrive access is ready and the full Services > Install folder is downloaded to Downloads\Install, the KR starter runs:


Downloads\Install\bootstrap\RUN_KR_BOOTSTRAP.ps1

That calls FirstLoginBootstrap -Apply -InstallMissingPackages -RunWindowsReadiness. The scripted contract is:

Area Scripted behavior
Codex communication CommunicationRuleCheck verifies communicate.md, SOA operator message folders, and first-read communication docs.
Codex workspace context WorkspaceOnboarding -WorkspaceId <id> -OpenWorkspace -OpenDocs can open the selected VS Code workspace plus the registered README/context/PlanMe first-read files.
File Explorer Bootstrap and post-bootstrap set File Explorer to show hidden files, protected files, and known file extensions for the signed-in Windows user.
Launchers First-login bootstrap installs the FRAME Desktop Builder shortcut on the Desktop and under Start Menu FRAME Install; taskbar pinning is best-effort and may still be manual on Windows 11.
PowerShell host Scripts prefer pwsh.exe when available and fall back to Windows PowerShell only for early bootstrap. Installing/upgrading PowerShell is handled by prerequisite/readiness checks, but the user should reopen terminals after an upgrade.
Browser/PDF defaults Windows readiness checks whether Edge is the browser default and Adobe Acrobat/Reader is the PDF default. Windows 11 protects default-app hashes, so the script reports and opens Default Apps when manual confirmation is needed rather than force-writing those associations.
Windows stability/readiness WindowsReadiness is exposed through the Install service router and is run during KR bootstrap. Its current implementation still delegates to projects\fr_install until that logic is fully promoted into Services\Install.

The important distinction is: profile-safe settings and FRAME launch surfaces are scripted; Windows-protected identity/default-app choices are checked and guided; final account sign-ins, MFA, Office/Adobe activation, and protected default-app confirmation remain operator-supervised.
