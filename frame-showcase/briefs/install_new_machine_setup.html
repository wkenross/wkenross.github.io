New Machine Setup Pattern

Purpose

FRAME treats a new Windows laptop as a reproducible automation workstation. The install lane prepares the applications, project folders, browser tooling, VS Code workspaces, and verification checks needed before workflow-specific automation begins.

The goal is not to clone a private machine. The goal is to rebuild the workstation pattern from reviewed files, synced project folders, and local runtime setup.

Operating Model

The setup separates synced project material from machine-local runtime state:

Layer Examples Portability
Synced project layer workspace files, Markdown plans, publication inputs, script source carried through OneDrive or Git
Machine tool layer PowerShell, Git, VS Code, Python, browsers, Office tools, BI tools installed per workstation
Runtime layer browser debug profiles, Office identity, WebDriver binaries, virtual environments, logs rebuilt locally

This keeps the new laptop aligned with the existing workstation while avoiding the unsafe practice of copying private browser sessions, credentials, tokens, cookies, or account recovery material.

Installer Posture

The preferred installer style is:

  • report first
  • apply only when explicit switches are supplied
  • tolerate already-installed tools
  • write verification output that another operator can review
  • keep passwords, tokens, and profile state out of scripts

Dry-run and readiness checks are part of the framework design. A command that changes the machine should make that clear in its name, parameters, and output.

Core Tool Readiness

A FRAME-ready Windows 11 laptop should verify:

  • PowerShell 7
  • Windows PowerShell compatibility for legacy scripts
  • Git and GitHub CLI
  • VS Code and required extensions
  • Python and the selected virtual environment
  • Microsoft Edge and Google Chrome
  • Edge WebDriver and PATH configuration
  • Acrobat Reader or Acrobat
  • Power BI Desktop where BI reporting is part of the lane
  • command-line helpers such as rg, jq, fd, 7zip, and package tooling

The install lane can stage installers, launch package-manager commands, and write follow-up instructions. Interactive sign-in, MFA, browser profile selection, and Office identity validation remain operator actions.

Project Folder Pattern

The framework supports both local and synced project roots. A typical workstation can use:


%USERPROFILE%\projects
%USERPROFILE%\OneDrive\projects

Workspace files should refer to environment variables wherever possible so the same workflow can run on a local-first desktop, a OneDrive-first desktop, or a temporary road-ready local sync.

Verification Checklist

Before treating a new machine as ready:

  • open the Install workspace in VS Code
  • run the install/update readiness script in report mode
  • confirm OneDrive project folders are present
  • confirm workspace files open without missing folder prompts
  • confirm Edge launches with the expected account profile
  • confirm WebDriver is available on PATH
  • confirm Python imports succeed for the core frame helpers
  • confirm the publisher can build a local wb_html site
  • confirm no private runtime folders were copied into public publish inputs

Public Safety

This public page intentionally avoids account names, passwords, tenant URLs, private browser folders, mailbox state, and local recovery details. Those details belong in private operator notes, not in the GitHub Pages showcase.
