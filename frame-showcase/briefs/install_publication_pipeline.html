Install Publication Pipeline

Purpose

The Install workspace now has a repeatable path for turning reviewed setup documentation into public website pages. The path keeps raw workspace notes separate from the generated GitHub Pages site.

Flow

The publication lane moves through five stages:

Stage Folder Role
intake wb_github\publish_inputs\workspaces\install\md reviewed Markdown candidates
manifest wb_github\publish_inputs\workspaces\install\manifests target path and review status
content wb_html\content\pages normalized site Markdown
generated HTML wb_html\html\briefs local rendered pages
root site wkenross.github.io\frame-showcase GitHub Pages output

This gives each operator a clear handoff point. Install authors stage pages. The HTML factory renders them. The GitHub publisher packages and pushes the result.

Publisher Command

The install lane is promoted with:


pwsh -NoProfile -ExecutionPolicy Bypass -File "$env:USERPROFILE\OneDrive\projects\wb_github\scripts\ps1\Publish-WbGithubSite.ps1" -PromoteInstallInputs -CommitAndPush -VerifyPublicUrls -CommitMessage "Publish install workspace showcase updates"

The same command can be run without -CommitAndPush when the operator wants to review generated HTML locally before publishing.

Safety Gate

The manifest is not a privacy review by itself. A page should enter the manifest only after the Markdown has been checked for:

  • passwords, tokens, cookies, and recovery material
  • account identifiers that do not belong in public copy
  • raw screenshots or mailbox state
  • private tenant URLs
  • customer workbook details
  • claims that overstate automation, certification, or tool capability

Relationship To Other Workspaces

The install publication lane complements workflow-specific lanes:

  • sr_bi_tools stages BI and reporting material
  • Search/update workspaces stage public-safe operational examples
  • resume and mobile workflows can contribute sanitized context and process

    • pages

  • wb_html remains the rendering factory
  • wb_github remains the publication control point

The result is a website that can show how FRAME is installed, operated, and extended without publishing the private working folders directly.

Acceptance Checks

  • Markdown candidate exists before manifest entry
  • manifest target path points to a briefs/*.html page
  • generated HTML builds locally
  • root GitHub Pages repo receives only generated public artifacts
  • live URL returns HTTP 200 after push and propagation
