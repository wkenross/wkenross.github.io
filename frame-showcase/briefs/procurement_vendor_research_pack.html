Procurement Vendor Research Pack

This brief summarizes public procurement-integration vocabulary used by the FRAME Procurement Adapter Workbench.

Working Summary

PunchOut is typically not JSON-first. The common enterprise vocabulary is cXML, with hosted catalog files, EDI, OCI, APIs, supplier portals, and middleware also appearing depending on the platform and customer.

FRAME can still normalize events into JSON internally:


cXML / hosted file / EDI / API input
  -> parser and validation check
  -> normalized adapter event JSON
  -> SQL/BI launch report
  -> Jira-ready backlog item

Discovery Questions

  • Which procurement platform owns the buyer session?
  • Is the customer asking for hosted catalog, PunchOut Level 1, PunchOut Level 2,

    • cXML PO/invoice, EDI, API, or a hybrid?

  • What identifiers matter: DUNS, NetworkID, supplier part number, UOM,

    • contract ID, buyer cookie, payload ID, account, ship-to, bill-to?

  • What is the test vs. production credential and signoff process?
  • Which events must be monitored after go-live?
  • Which exceptions become customer support cases, product backlog items, or

    • technical defects?

Prototype Acceptance Checks

  1. Hosted catalog validates required product and price fields.
  2. PunchOut setup request parses buyer, supplier, endpoint, payload, and

    3. session metadata.

  3. Setup response returns a synthetic landing URL.
  4. Cart return creates a normalized requisition event.
  5. PO, confirmation, invoice, and ASN fixtures produce SQL-ready rows.
  6. Missing fields generate readable exceptions and Jira-ready follow-up.
  7. Launch report groups readiness by project plan, setup/testing, go-live, and

    8. optimization.
