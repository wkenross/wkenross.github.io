WB HTML Publish Workflow PlanMe

Updated: 2026-06-09

Purpose

Define the review-first workflow for turning generated HTML into a GitHub-hosted site.

This workflow is the publication lane for reviewed FRAME evidence. It should only render material that has passed the relevant ObfuscateMe, TrainMe, and source-review checks in the owning sr_ workflow.

Recommended First Publishing Path

Start with GitHub Pages from a repository folder:


docs/

or a dedicated branch:


gh-pages

For the first pass, keep the site local or private until the source catalog and generated pages pass privacy review.

Build And Review


py "$env:USERPROFILE\OneDrive\projects\wb_html\scripts\py\build_public_data_catalog.py"
py "$env:USERPROFILE\OneDrive\projects\wb_html\scripts\py\build_site.py"
ii "$env:USERPROFILE\OneDrive\projects\wb_html\html\index.html"

Review before publishing:

  • generated page titles
  • broken links
  • private names and paths
  • screenshots and image metadata
  • certification claims
  • vendor tenant details
  • .pbix, workbook, PDF, and email attachment safety
  • public-data attribution and terms of use
  • SQL/Power BI outputs use public or synthetic data

FRAME Evidence Pipeline


local fr_ workflow
  -> ObfuscateMe / declutter / no-scrape guard
  -> OneDrive sr_ workflow
  -> sr_sql_data SQL-ready load and validation
  -> Power BI / CSV reporting evidence
  -> wb_html Markdown-to-HTML rendering
  -> private GitHub Pages review
  -> selected public pages

wb_html should not be used to launder private material into a public site. It is the last review surface, not the first cleanup surface.

Publication Options

Option Use When
local html folder drafting and private review
private GitHub Pages limited reviewer audience
public GitHub Pages only after scrub and approval
MkDocs later site grows beyond simple static pages
Docusaurus later versioned docs and heavier navigation become useful

GitHub Pages Shape

If publishing from /docs, copy the generated files:


wb_html/html/* -> repository/docs/*

If publishing from gh-pages, copy generated files into that branch only after review.

Done Criteria

  • local HTML builds
  • review checklist passes
  • generated output is separate from source Markdown
  • source catalog and public-data catalog are generated
  • publication target is private by default
  • public release has no private data or unearned credential claims

Product Risk Analysis

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.
