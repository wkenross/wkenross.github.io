WB HTML Style Guide

Updated: 2026-06-30

Purpose

Give FRAME web pages a consistent style while allowing different document types to feel distinct.

The visual tone should be calm, credible, and evidence-oriented. It should feel like an operations and data-product workbench, not a glossy marketing page.

Page Kinds

Kind Use Visual Treatment
planme strategy, roadmap, decisions blue accent, roadmap label, larger section headings
trainme practical learning, step-by-step work green accent, numbered routines, instructional tone
certifyme readiness, credentials, evidence amber accent, checklist and evidence table emphasis
resume role-specific career material compact typography, tight spacing, proof bullets
tool Jira, Salesforce, HubSpot, Power BI neutral accent, integration and safety notes
default general docs quiet neutral treatment

Artifact Types

site_manifest.json can also set artifact_type for pages that need a public-facing document label. This is separate from kind: kind preserves FRAME operator language, while artifact_type tells product, Agile, architecture, and executive readers what kind of artifact they are viewing.

Pilot values:

Artifact Type Use
overview Service or product landing page.
architecture Boundaries, interfaces, data contracts, risks, and decisions.
business-plan Product brief, personas, value hypothesis, metrics, and proof status.
roadmap Sequencing, milestones, horizons, and tradeoffs.
backlog Epics, user stories, acceptance criteria, sprint candidates, and risks.
runbook Operating steps, validation, launch checks, rollback, and support evidence.
research Source-backed discovery, vendor notes, assumptions, and open questions.
tool Workbench, fixture, or reusable service artifact.

Artifact type renders as a quiet metadata pill and a restrained page accent. Pages without artifact_type build normally.

Shared Rules

  • Use one restrained FRAME style across Jira, Salesforce, HubSpot, and Power BI

    • pages.

  • Do not imitate vendor dashboards or account screens.
  • Use generic tenant/account wording unless a private reviewer explicitly needs

    • the real path.

  • Prefer tables, checklists, and short evidence sections.
  • Keep pages useful when printed or saved as PDF.
  • Avoid screenshots until they are scrubbed.

Layout

  • left navigation on desktop
  • top navigation on narrow screens
  • readable content width
  • no nested cards
  • no decorative gradient blobs
  • code and commands in clearly framed blocks
  • tables scroll on small screens

Document Conversion Notes

PlanMe pages should lead with purpose, goals, non-goals, phases, and acceptance criteria.

TrainMe pages should lead with why a package or approach was chosen, then show the standard loop, commands, practical lessons, and done criteria.

CertifyMe pages should lead with credential target, official references, evidence map, readiness checklist, and truthful resume language.

Product Risk Analysis

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.

Publication rule: examples use public-safe or synthetic data only. They do not connect to live websites, mailboxes, CRM systems, marketplace accounts, or procurement tenants.
