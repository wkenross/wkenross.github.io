WB HTML TrainMe

Updated: 2026-06-09

Purpose

Teach the practical workflow for turning FRAME Markdown into a clean static site.

The core habit is simple: write in Markdown, curate for safety, build HTML, review locally, then publish only the reviewed output.

Why This Approach

Static HTML is the first choice because it is:

  • easy to host on GitHub Pages
  • easy to inspect before publishing
  • independent of a running server
  • friendly to Markdown source files
  • safer than wiring live tools into a public site

The builder starts with Python standard library code so the workflow can run on a fresh FRAME workstation without a front-end package install.

Standard Loop

  1. Choose a source document from a workflow folder.
  2. Decide whether it is public, limited, internal, or blocked.
  3. Copy only reviewed ideas into content/pages.
  4. Pick a page kind: planme, trainme, certifyme, resume, tool, or

    5. default.

  5. Add the page to config/site_manifest.json.
  6. Run scripts/py/build_site.py.
  7. Open html/index.html.
  8. Review links, titles, tone, and private references.
  9. Publish only after review.

Practical Lessons

  • A website is a packaging layer, not the source of truth.
  • Markdown stays close to the workflow that owns the work.
  • Generated HTML should be disposable.
  • A good showcase page explains evidence, limits, and next steps.
  • Tool pages should show transferable workflow thinking, not private tenant

    • state.

Useful Commands


py "$env:USERPROFILE\OneDrive\projects\wb_html\scripts\py\build_site.py"
ii "$env:USERPROFILE\OneDrive\projects\wb_html\html\index.html"

Review Checklist

  • no private URLs
  • no real customers or contacts
  • no tokens or secrets
  • no private workbook names
  • no screenshots with account details
  • no claims of certifications not yet earned
  • all data examples are synthetic or clearly scrubbed

Done Criteria

  • The site builds locally.
  • Navigation works.
  • Each page has a visible kind label.
  • Review status is represented in the page metadata or source catalog.
  • The generated output can be deleted and rebuilt.

Product Risk Analysis

How FRAME Reads Procurement Risk

FRAME treats procurement risk as a product discovery funnel, not just a technical integration checklist.

Sources
Public vendor documentation, supplier enablement terms, procurement learning material, cXML and PunchOut references, Sigma-Aldrich eProcurement content, and synthetic FRAME fixtures.
Research Approach
Each provider lane is mapped by access model, setup path, transaction lifecycle, likely owner, open question, and claim boundary. No private tenant access is implied.
Risk Reporting
Risks are reported as drop-off stages: access, supplier master, catalog discovery, PunchOut auth, cart return, approval, PO, invoice/ASN, support, and adoption.
Dashboard Integration
The risk map feeds partner matrices, SWOT, sprint backlog items, launch-health metrics, and BI-ready dashboard signals such as cart return exceptions and PO-to-invoice lag.

Publication rule: examples use public-safe or synthetic data only. They do not connect to live websites, mailboxes, CRM systems, marketplace accounts, or procurement tenants.
